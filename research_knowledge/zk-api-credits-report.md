## The constructions

### ZK API Usage Credits (the main post)

Registration: user picks secret k, computes identity commitment Hash(k), deposits D into a contract that inserts the commitment into an on-chain Merkle tree. The deposit is actually two buckets, D + S: D is the RLN stake, claimable by anyone who can present the recovered secret after a double-signal; S is a "policy stake" the server can burn (not claim, to kill the false-accusation profit motive) via slashPolicyStake(Nullifier, ViolationEvidence) for ToS violations (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104).

Per request, the user picks the next ticket index i (a strictly increasing counter 0, 1, 2, ...) and produces a zk proof (post says ZK-STARK) of: (1) Merkle membership; (2) refund summation R = Σ v_j over server-signed refund tickets; (3) the solvency invariant "(i + 1) · C_max ≤ D + R"; (4) the RLN components with the epoch replaced by the ticket index: slope a = Hash(k, i), signal x = Hash(M), y = k + a·x, Nullifier = Hash(a). The word "epoch" does not appear in the post; the ticket index plays the role RLN's epoch plays, converting a rate limit into a spend-count limit (verified against the thread print view, https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print; RLN reference: https://rate-limiting-nullifier.github.io/rln-docs/rln.html).

What the provider stores: a local "Spent Tickets" database of seen nullifiers. Verbatim: "The Server checks the Nullifier in its 'Spent Tickets' database: Fork/Double-Spend Check: If the Nullifier exists with a different x (Message), the user tried to spend the same ticket on two different requests," in which case two points on the same line let it solve for k and slash D on-chain. The post says nothing about sharing this set with anyone, and the thread never discusses multiple providers sharing a deposit or a spent set (checked explicitly against the print view).

Refunds (the hard part): C_max is escrowed per ticket but actual cost varies wildly, so after serving, the server issues a signed refund ticket r = {v, sig} with v = C_max − C_actual. Variant 1: the user accumulates tickets and proves their sum in-circuit. Variant 2: the server keeps a homomorphically encrypted running total, E(R_new) = E(R) ⊕ E(r), and signs the ciphertext; after omarespejel showed the static E(R) is linkable across settlements (simulation gist: https://gist.github.com/omarespejel/c3f4f2aa12b1de10467601d77d0e6232), dcrapis updated the spec to require re-randomization with proof of equivalence (BBS+/blind signatures) and retracted v2 parallelization. Vitalik's own reply explains why refunds can't just be dropped for LLMs: "with no refunds my overhead would be like 100x ... I need to have a >$5 budget to make requests ... but on average each request costs like a cent," with cost following "some infinite-variance Levy-ish distribution" (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print).

Unlinkability claim and adversary: against the server (and chain observers post-deposit), "honest users who stay within protocol limits remain unlinkable," prompts "remain unlinkable to the user identity," and requests "cannot be linked to the same depositor (or to each other)." Stated/thread-established limits: (a) deposit-time linkability, dbrizz: "That deposit is a public on-chain transaction linking the user's address (and identity, via exchange KYC trails) to the contract," fix proposed = shielded deposits / Privacy Pools-style proof of innocence; (b) traffic-analysis relinking via token counts, TTFT, latency, speculative-decoding acceptance (omarespejel claims ~96% relinkage in simulation; WGlynn: "the server sees token counts, latency, timing, and prompt structure. That's a richer fingerprint than the wallet address itself"); (c) refund trust: the anonymous user cannot dispute an under-reported C_actual without deanonymizing; (d) self-slashing race (4aelius, sergei-tikhomirov): a user who has consumed service the provider hasn't yet claimed can deliberately double-sign, racing the provider to its own deposit; unresolved in the thread; (e) prompt-content stylometry, acknowledged as a separate layer (all: https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print).

### Is it "fundamentally a zk payment channel bound to one recipient"? Yes, and a thread commenter says so almost verbatim

sergei-tikhomirov (Feb 27, ex-Lightning): "Concretely, we can view the payment mechanics here as a payment channel ... We want value transfer from the user to the provider proportionally to the amount of service provided, without on-chain transaction per each request. The provider must be sure that at any time it can claim on-chain the fair share of the user's deposit" (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print). The channel mapping is precise:

- Channel open = the deposit D into the Merkle-tree contract.
- Channel state = the triple (D on-chain, monotone ticket index i, server-signed refund total R). The solvency circuit "(i+1)·C_max ≤ D + R" is exactly a channel balance invariant; each request is a state update the user proves valid in zk and the server "countersigns" by issuing the refund ticket.
- What makes it zk: the provider verifies balance updates without learning which channel (which member) it is updating. It is a payment channel where the payer side of every update is anonymous within the deposit set.
- Recipient binding: R is a sum of tickets signed by one server key, and the spent set is that server's local database. Refund credit earned at server A is unspendable at server B, and B cannot detect indices already spent at A. So one deposit is safely usable at exactly one provider; N providers means N deposits (N× capital) unless nullifier sets are shared. The post and thread are silent on multi-provider use (verified, print view above).
- Close/settle: this is the construction's soft spot. The provider claims its earned share of D periodically; the post does not pin down when, and sergei-tikhomirov shows the gap: claim-every-request defeats the point, claim-every-10 creates the self-slash race ("what if the user makes 9 requests, and then deliberately double-signs and tries to slash its own deposit?"). MicahZoltu also answered JohnGuilding's "could a simpler design be a state channel" with the key differentiator: "A state channel would certainly be simpler, but it would still correlate requests with each other" (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print).

### PrivateX402 (the linked follow-up, and the wrong kind of channel for you)

McMenamin & Grigor extend x402 machine payments: user commits Leaf = H(SessionKey, AgentAddress, MaxSpend) per recipient into a Merkle tree, deposits total funding plus 50% collateral, and proves in zk that allocations sum to the deposit; "the contract stores only the root and total funding — individual allocations remain private." Payments are plain EIP-191 signed cumulative receipts (no per-payment proof); settlement posts XOR-encrypted per-agent amounts, recipients dispute within a window by revealing session keys, fraud burns collateral, and "agents batch-claim across all channels settled within an epoch" to hide per-channel breakdown. Proofs are currently TEE-emulated with zk planned (https://ethresear.ch/t/privatex402-privacy-preserving-payment-channels-for-multi-agent-ai-systems/24151).

The crucial contrast: PrivateX402 is multi-recipient from one deposit ("N user-agent channels ... 1 set-up and 1 settlement transaction per user") but its privacy target is on-chain observers and other recipients, not the payee. The recipient sees a per-channel session key and cumulative receipts, so all requests within one user-recipient channel are linkable to each other by the recipient. It explicitly positions against ZK API Credits: per-payment-zk designs need "N deposits/commitments" for N providers, while PrivateX402 "covers N agents with a single deposit and keeps payments proof-free ... concentrating proofs at setup/settlement/claim." Both posts admit the deposit-boundary ceiling: transparent stablecoin funding caps the privacy of either design (same URL).

## Relevance to our egress fleet (opinionated)

The one-recipient binding in ZK API Credits is not cryptographic doctrine, it is two specific engineering facts: (1) refund tickets are signed by one provider key, (2) the spent set is one provider's local DB. For a Tor egress fleet you can delete fact (1) entirely: egress requests are near-uniform cost, unlike LLM inference. Vitalik's whole 100x-overhead argument for refunds (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print) does not apply to "fetch one URL." Drop refunds, price every ticket at flat C, and the construction collapses to: prove membership + "(i+1)·C ≤ D" + per-index RLN nullifier. That is a strict superset of the Semaphore+RLN stack we already run; the only new circuit constraint over our epoch-RLN is the counter inequality, and the nullifier domain changes from epoch to index (you likely want both: index-nullifier for spend, epoch-nullifier for rate).

That collapse is what decides your (a)-vs-(b) question in favor of (a), the distributed spent set, and the thread accidentally tells you why: RLN slashing makes the spent set eventually-consistent-safe. In Chaumian e-cash, gateways would need a synchronous shared spent set because a double-spend is otherwise free. Here a cross-gateway double-spend (same nullifier, different x, at gateways A and B) is detected at reconciliation, k is recovered, and D is slashed, so the async window is priced, not trusted. Gateways gossip (nullifier, x, y) tuples on whatever lag they like; the economic bound on exposure is D. Nullifiers are unlinkable, so the gossip leaks only global volume/timing across gateways, which mutually distrusting gateways can tolerate. Caveat you must engineer around: the self-slash race (sergei-tikhomirov/4aelius, unresolved in the thread) means gateways should claim earned deposit frequently, or serve only up to a small unclaimed balance per epoch, because an anonymous user racing its own slash is otherwise a rational strategy.

Option (b) as literally instantiated by the literature is wrong for our threat model. PrivateX402's per-pair channels are cheap and multi-recipient, but the recipient sees a stable session key: every request a member sends through gateway G is linkable by G into one profile (https://ethresear.ch/t/privatex402-privacy-preserving-payment-channels-for-multi-agent-ai-systems/24151). Our adversary is the gateways. A per-member-per-gateway "zk payment channel" in the ZK-API-Credits sense (anonymous within the member set even to the paid gateway) does work, but costs N× deposits per member across N gateways and hands each gateway an isolated spend history anyway. The only reason to prefer (b) is if you cannot stomach nullifier gossip between gateways; you can, because the tuples are already anonymous.

Two imported warnings that survive the port to Tor: first, deposit-edge linkability (dbrizz): the on-chain deposit KYC-links a member to the membership set, so fund D through a shielded pool or Privacy Pools-style mechanism or the anonymity set is a fiction at the edges (https://ethresear.ch/t/zk-api-usage-credits-llms-and-beyond/24104/print). Second, WGlynn's point that payment unlinkability is bounded by traffic fingerprints applies to egress too (destination, response size, timing), though Tor circuits give us more cover than an HTTPS API does; his quantized-pricing-tiers suggestion is exactly the flat-ticket design above. Also steal the dual-stake idea wholesale: D slashed cryptographically for double-spends, S burnable-not-claimable by gateways for abuse-policy violations, which fits an egress fleet's abuse problem better than it fits LLM billing.
